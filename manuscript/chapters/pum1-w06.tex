\chapter{Obiekty i Interfejsy (Wykład 6)}

W poprzednim rozdziale omówiliśmy klasy i dziedziczenie. Kotlin wprowadza jednak kilka unikalnych konstrukcji, które znacznie upraszczają typowe wzorce projektowe, takie jak Singleton czy anonimowe klasy wewnętrzne. Mowa o słowie kluczowym \texttt{object}.

\section{Słowo kluczowe \texttt{object}}

W Kotlinie \texttt{object} to specjalna konstrukcja, która \textbf{jednocześnie definiuje klasę i tworzy jej jedyną instancję}. Jest to realizacja wzorca \textbf{Singleton} (wbudowana w język).

Gdy chcemy mieć pewność, że w całej aplikacji istnieje tylko jeden egzemplarz danej klasy (np. konfiguracja, menedżer bazy danych), używamy deklaracji obiektu:

\begin{minted}[frame=single,framesep=10pt]{kotlin}
object DatabaseConfig {
    const val DB_URL = "jdbc:mysql://localhost:3306/db"
    var connectionCount = 0

    fun connect() {
        connectionCount++
        println("Connected to $DB_URL")
    }
}
\end{minted}

Do składowych takiego obiektu odwołujemy się bezpośrednio przez nazwę (bez tworzenia instancji operatorem \texttt{()}):

\begin{minted}[frame=single,framesep=10pt]{kotlin}
DatabaseConfig.connect()
println(DatabaseConfig.connectionCount)
\end{minted}

Obiekty te są inicjalizowane \textbf{leniwie} (lazy) – dopiero przy pierwszym odwołaniu się do nich.

Czasami potrzebujemy obiektu danej klasy (lub implementującego interfejs) tylko w jednym miejscu (\textit{ad-hoc}), bez konieczności nadawania mu nazwy. W Javie używalibyśmy anonimowych klas wewnętrznych. W Kotlinie używamy wyrażeń obiektowych (\texttt{object expression}).

\begin{minted}[frame=single,framesep=10pt]{kotlin}
interface MouseListener {
    fun onClick()
}

// Tworzymy obiekt anonimowy implementujący interfejs
val listener = object : MouseListener {
    override fun onClick() {
        println("Kliknięto myszką!")
    }
}
\end{minted}

Możemy też dziedziczyć po klasie i implementować wiele interfejsów jednocześnie:
\begin{minted}[frame=single,framesep=10pt]{kotlin}
open class A(x: Int)
interface B

val ab = object : A(1), B {
    val y = 15
}
\end{minted}

Od nowszych wersji Kotlina, obiekty (\texttt{object}) mogą być oznaczane jako \texttt{data object}. Działa to analogicznie do \texttt{data class} – generuje czytelną metodę \texttt{toString()}.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
data object EmptyState

println(EmptyState) // Wypisze "EmptyState" zamiast "EmptyState@1b2c3d"
\end{minted}

Jest to szczególnie przydatne przy \textbf{Sealed Classes/Interfaces}, gdzie część wariantów to klasy z danymi, a część to proste bezstanowe obiekty.

\section{Interfejsy}

Interfejs definiuje kontrakt – zestaw metod i właściwości, które klasa implementująca musi dostarczyć. W Kotlinie interfejsy są bardzo elastyczne: mogą zawierać metody abstrakcyjne oraz metody z domyślną implementacją.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
interface Drivable {
    val maxSpeed: Int // Właściwość abstrakcyjna
    
    fun drive() // Metoda abstrakcyjna (brak ciała)
    
    // Metoda z domyślną implementacją
    fun stop() {
        println("Zatrzymuję się.")
    }
}
\end{minted}

Klasa implementująca interfejs używa tej samej składni co przy dziedziczeniu (\texttt{:}):

\begin{minted}[frame=single,framesep=10pt]{kotlin}
class Car : Drivable {
    override val maxSpeed = 200 // Musimy nadpisać właściwość
    
    override fun drive() {
        println("Jadę autem z prędkością $maxSpeed")
    }
    // Metody stop() nie musimy nadpisywać, użyje domyślnej
}
\end{minted}



Interfejsy w Kotlinie nie mogą przechowywać \textbf{stanu} (nie mają pól instancji). Mogą jednak deklarować właściwości:
\begin{itemize}
    \item Abstrakcyjne (klasa musi je nadpisać).
    \item Zdefiniowane akcesorami (getter), o ile nie odwołują się do \texttt{field}.
\end{itemize}

\begin{minted}[frame=single,framesep=10pt]{kotlin}
interface Named {
    val name: String // abstrakcyjne
    
    val nameLength: Int
        get() = name.length // OK, oblicza na podstawie 'name'
}
\end{minted}

Klasa może implementować wiele interfejsów jednocześnie. Jeśli pojawi się konflikt nazw metod (dwa interfejsy mają metodę o tej samej sygnaturze i obie mają domyślną implementację), musimy jawnie wskazać, którą wersję wywołać, używając \texttt{super<Typ>}.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
interface A {
    fun foo() { print("A") }
}
interface B {
    fun foo() { print("B") }
}

class C : A, B {
    override fun foo() {
        super<A>.foo() // Wywołaj wersję z A
        super<B>.foo() // Wywołaj wersję z B
    }
}
\end{minted}

\section{Zastosowania Interfejsów}

Interfejsy w programowaniu obiektowym pełnią kluczową rolę, wykraczającą poza proste definiowanie metod.

Interfejs tworzy nowy \textbf{typ danych}, którym możemy posługiwać się w kodzie. Zmienna, parametr funkcji czy pole w klasie może być typu interfejsu. Dzięki temu kod staje się niezależny od konkretnej implementacji.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
fun playMusic(player: Playable) { // Parametr jest typu interfejsu
    player.play()
}
\end{minted}

Interfejs narzuca klasom implementującym obowiązek dostarczenia kodu dla zdefiniowanych metod. Gwarantuje to, że każdy obiekt \textit{spełniający kontrakt} (implementujący interfejs) posiada określone zachowania.

Dzięki traktowaniu interfejsu jako typu, możemy uzyskać \textbf{polimorfizm}. Obiekty różnych klas (np. \texttt{Car}, \texttt{Bike}, \texttt{Plane}) mogą być traktowane w jednolity sposób (jako \texttt{Drivable}), o ile implementują ten sam interfejs. Pozwala to na pisanie uniwersalnego kodu, operującego na abstrakcjach, a nie konkretach.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
val vehicles: List<Drivable> = listOf(Car(), Bike())
for (v in vehicles) {
    v.drive() // Zadziała dla każdego obiektu, niezależnie od konkretnej klasy
}
\end{minted}

Czasami interfejs nie posiada żadnych metod ani właściwości, jest to tak zwany \textbf{interfejs marker} (interfejs znacznikowy). Służy wtedy jedynie do \textbf{oznaczenia} klasy, by nadać jej pewną cechę lub poinformować framework o jej specjalnym traktowaniu.
Przykładem w Javie/Kotlinie jest \texttt{Serializable} (oznacza, że obiekt można serializować) lub \texttt{Cloneable}. Taki interfejs jest często używany w frameworkach, które muszą wiedzieć, że dana klasa może być używana w określonym kontekście.

Interfejsy pozwalają nadać wspólną cechę klasom, które nie są ze sobą powiązane w hierarchii dziedziczenia. Np. interfejs \texttt{Comparable} sprawia, że obiekty można porównywać (i sortować), a \texttt{Iterable} pozwala iterować po obiekcie w pętli \texttt{for}.

\section{Interfejsy Funkcyjne (SAM)}

Jeśli interfejs posiada \textbf{dokładnie jedną metodę abstrakcyjną}, nazywamy go interfejsem funkcyjnym lub \textbf{SAM} (Single Abstract Method).

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.9\textwidth]{fig/pum1-w06-interfejsy.png}
    \caption{Interfejsy w Kotlinie.}
    \label{fig:interfejsy}
\end{figure}

W Kotlinie oznaczamy takie interfejsy słowem \texttt{fun interface}.

\begin{minted}[frame=single,framesep=10pt]{kotlin}
fun interface Predicate {
    fun accept(i: Int): Boolean
}
\end{minted}

Dzięki temu możemy używać konwersji SAM – zamiast tworzyć obiekt anonimowy, możemy przekazać \textbf{wyrażenie lambda}:

\begin{minted}[frame=single,framesep=10pt]{kotlin}
// Tradycyjnie (obiekt anonimowy)
val p1 = object : Predicate {
    override fun accept(i: Int) = i > 0
}

// Z użyciem konwersji SAM (lambda)
val p2 = Predicate { it > 0 }
\end{minted}

To znacznie skraca kod, zwłaszcza przy obsłudze zdarzeń w UI (np. \texttt{OnClickListener}).

\section{Klasa Abstrakcyjna vs Interfejs}

Skoro interfejsy mogą mieć metody z implementacją, czym różnią się od klas abstrakcyjnych?

\begin{enumerate}
    \item \textbf{Stan}: Klasy abstrakcyjne mogą mieć stan (pola, zmienne instancji). Interfejsy nie mogą (tylko właściwości bez backing field).
    \item \textbf{Konstruktory}: Klasy abstrakcyjne mają konstruktory, interfejsy nie.
    \item \textbf{Wielodziedziczenie}: Klasa może dziedziczyć tylko po jednej klasie (abstrakcyjnej lub nie), ale może implementować \textbf{wiele} interfejsów.
\end{enumerate}
